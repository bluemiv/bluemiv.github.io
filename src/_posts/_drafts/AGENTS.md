# Post Writing Rules

이 폴더의 MDX 글을 작성하거나 수정할 때는 아래 기준을 지킨다.

## 기본 원칙

- 프론트매터 필수 필드(`title`, `description`, `createdAt`, `updatedAt`, `tags`, `release`, `author`, `thumbnail`)를 모두 작성한다.
- 본문이 실제로 작성된 글만 `release: true`로 둔다. 프론트매터만 있는 스텁 글은 `release: false`를 유지한다.
- 작성자는 사이트 프로필과 맞춰 `Taehong Kim`을 사용한다.
- 제목은 본문에서 다시 `#`로 쓰지 않고, 본문은 `##`부터 시작한다.

## SEO와 신뢰도

- 검색 엔진보다 독자를 먼저 생각해서, 문제 상황과 독자가 얻는 결론을 글 앞부분에 명확히 둔다.
- 단순 개념 정리만 하지 말고, “실무에서 자주 하는 실수”, “내가 사용하는 기준”, “주의할 점” 중 하나 이상을 포함한다.
- 기술 설명에는 가능한 한 공식 문서나 1차 출처 링크를 1개 이상 포함한다.
- 내부에서 이어지는 공개 글이 있으면 자연스러운 문장 안에 링크한다. 아직 `release: false`인 글에는 링크하지 않는다.
- 링크 앵커는 “여기”보다 글의 주제를 드러내는 문구를 사용한다.
- 외부 링크는 렌더러에서 새 탭으로 열리므로, 본문에서는 URL을 정확히 쓰고 불필요한 HTML 속성을 직접 넣지 않는다.
- `description`은 기능 나열보다 독자가 글에서 얻는 판단 기준이나 해결할 문제를 드러내도록 작성한다.

## 코드와 예제

- 코드 블록에는 언어를 반드시 지정한다.
- 예제는 복사해서 이해할 수 있을 만큼 완결성 있게 작성한다.
- Spring DI 예제처럼 설계 의도를 보여주는 글에서는 구체 클래스 직접 의존보다 인터페이스 기반 예제를 우선한다.
- 민감 정보 예시는 실제 값처럼 보이지 않게 작성하고, 운영 환경에서는 환경변수나 Secret 사용을 안내한다.

## 이미지와 공개 전 확인

- `thumbnail` 경로에 실제 이미지 파일이 있는지 확인한다.
- 썸네일은 현재 블로그 카드/OG 메타 기준인 1080x589 비율을 유지하고, 글 제목을 설명하는 장면으로 만든다.
- 새 글을 공개할 때는 `pnpm build`로 MDX 파싱과 정적 경로 생성을 확인한다.
- 날짜 메타데이터와 JSON-LD가 유효한 ISO 문자열로 출력되는지 확인한다.
- 공개 전 대표 글 1개는 `og:title`, `og:description`, `og:image`, `twitter:image`, `BlogPosting`, `BreadcrumbList`가 출력되는지 확인한다.
